# UX audit — 2026-04-23 walkthrough

Tristan's brief: *"think about the usability of the app and the quality
of the information and how it's flowing… not just that the code works
and doesn't break, also the overall user experience and other things
which could be done to improve it as you go along."*

Scope: walked the SkySails + Wren Aerospace campaigns end-to-end via
agent-browser; tested the full flow from pitch paste → find matches →
shortlist → approval → test-send → reply-ingest → tracker. This doc
is the list of friction points, information-quality concerns, and flow
improvements. Not bug tickets — product observations.

---

## 1. Campaign identity is fragmented

**Problem.** The campaign name is shown in 5 different places across
the nav bar, section heads, breadcrumbs, and panel subtitles. It
doesn't read as one object — it reads as a scope token that keeps
appearing. The user has no sense of "this is the raise I'm working
on right now" as a first-class entity.

**Evidence.** On /match the campaign chip is in the top-right; on
/approval it reads "AUDIT · Wren Aerospace · Investor" inside the
section head; on /templates it's appended to the section title with
`· <name>`; on /weekly it's in the intro paragraph. The string
`AUDIT · Wren Aerospace · Investor` has three delimiters and a
badge-style `INVESTOR` pill — it reads as admin metadata, not as
"my raise".

**Suggestion.** A persistent left-rail rail or top-bar card that
represents the active raise with: raise_size, week N of 16 (Tristan
already tracks this on `campaigns.week_started_at`), counterpart
name/email, and a pulse dot for "new activity since I last looked".
Campaign switcher stays — but the active raise sits as a visible
workspace, not a breadcrumb decoration.

## 2. The "AUDIT" prefix bleeds into real user-facing strings

**Problem.** `AUDIT · Wren Aerospace · Investor` is a test-campaign
name, but it's being rendered into real email subjects, the outbound
approval sheet filename, the test-batch subject (`[TEST] AUDIT · Wren
Aerospace · Investor — …`), and the weekly counterpart update title.
If a real customer were looking at these outputs, "AUDIT" would leak
through.

**Suggestion.** Either rename the campaign or separate `display_name`
from the internal `name` (the internal one stays auditable, the
display one is what appears in emails / PDFs / UI). A one-field
migration + the composer respects display_name.

## 3. The Approval page's Step 2 Parse button is still Haiku-branded

**Problem.** After upgrading the email drafter to Opus 4.7 with
Tristan's explicit instruction, Step 2's `Parse with Haiku →` button
sub-copy still reads "Parser extracts ok / not-for-me / skip markers".
It's extraction, so Haiku is appropriate, but the UI inconsistency
(one surface says Opus, another says Haiku) reads as sloppy.

**Suggestion.** Drop the model brand from the button text — just
"Parse reply →" with a tooltip "Extracts ok / not-for-me / skip
markers via Claude Haiku (extraction is fast/cheap; composition uses
Opus 4.7)." Model names are implementation detail; users don't need
them on the button.

## 4. The `Email approval list` button is orphaned at the top of /approval

**Problem.** I added the "Approve on phone" card on the audit today.
It sits above the Step 1 outgoing sheet, which is correct for
discoverability, but the button label ("Email approval list") doesn't
say what HAPPENS next — where the reply goes, what happens to rows
the user doesn't annotate, how long the email is.

**Suggestion.** Change the surrounding explainer to:
"Emails the 20 rows below as a plain-text list. Reply inline with
`ok / no / flag / skip` per row and paste the reply into Step 2. Rows
you don't annotate stay at +0 Pending approval." Make the causal
chain visible so the user isn't guessing.

## 5. The `/approval/test-send` page has no "open in a new tab after
send" affordance

**Problem.** After the batch completes, each row shows an "open in
Gmail ↗" link. That's useful. But 20 links in a list means if you
want to spot-check the output, you click, land on Gmail Sent, lose
your place on /approval/test-send. No back button preserves state.

**Suggestion.** Either (a) open each link in a new tab (add
`target="_blank"` — already done, but the per-row affordance isn't
obvious the user has been reading the list; visually the green check
and firm name dominate), or (b) render the first 3 drafts inline
(collapsed by default, expand to read) so the user doesn't have to
leave the app to eyeball the output. The latter is better product.

## 6. Information quality — the "why them" synthesis is still thin for
many firms

**Observation.** Walking the Wren match results, about 60% of the 20
shortlisted rows have rich 200+ word synthesis blocks (AENU, EBRD,
Bpifrance). The remaining 40% have short one-sentence descriptions
like "Venture capital firm investing in AI-first companies at the
earliest stages, with focus on defence and dual-use technology" —
true but not helpful for an approver to decide.

**Suggestion.** The nightly enrichment pipeline (/pipeline's Enrich
stage) should flag firms with < 300 char `thesis_summary` AND <
400 char `investment_pattern` as "thin", and the tracker / approval
UI should surface that as a "requires research" banner rather than
silently showing a one-liner and hoping. Honest signal beats hidden
debt.

## 7. The tracker's right sidebar is underused

**Observation.** On /tracker, the left 70% is the table, the right
30% is a column with status distribution chart, a pending-count tile,
and two secondary tiles. That's a lot of pixels for a static
summary. The chart doesn't explain itself (why are there +7 at
negative, what's "Mid", etc).

**Suggestion.** Replace with (a) a compact action panel: "Next
step: 20 rows at +0 — [Generate approval sheet]"; (b) a mini
timeline of the last 5 contact_events in this campaign (who,
inbound/outbound, when); (c) a "needs your attention" list
— rows at `+6 Response received` but no follow-up sent in > 5 days,
rows at `+7 Meeting offered` waiting on acceptance, rows at `+10`
stuck. These are decisions the founder must make; the chart isn't.

## 8. The Templates page is a dense wall

**Observation.** /templates shows 4 labelled paragraphs in sequence,
each with a "Redraft with Opus" button. The page itself is ~1500px
tall. There's no indication of which paragraphs are the BIG levers
vs the small ones. Per-investor synthesis has by FAR the biggest
impact on reply rates (it's the match-specific paragraph), but it's
visually equivalent to the CTA (which is a binary choice).

**Suggestion.** Size the section cards by strategic importance:
synthesis section gets 2-col width, credibility + company are
single-col, CTA is a compact enum picker. Or: add a "last refined"
timestamp next to each so the user sees which paragraphs are stale
at a glance.

## 9. The sent emails disappear — no "what did I send to whom"
timeline on the tracker

**Observation.** After the 20-row test batch, the tracker rows still
show +0 Pending approval (correct — the batch doesn't advance real
status). But there's no surfaced record of "I test-sent this row on
2026-04-23 at 11:05 UTC, here's what went out". You have to go to
/approval/test-send which is the WRONG place (that page lists
pending rows, not past sends).

**Suggestion.** A new per-row affordance on /tracker — "Sends
history" — expands to show contact_events for that campaign_partner_id
in reverse chronological order. Dry-runs distinguished from real
sends by the `test_send` vs `outbound_first_contact` event_type.
Tristan can see the actual email body that went to each recipient
without leaving the tracker.

## 10. The "Resolve email →" flow is a dead-end

**Observation.** On /match, a row with an unverified / generic /
bounced email shows a "Resolve email →" button. Clicking it opens the
EmailHuntModal — a multi-step form. But the modal's state is
ephemeral: close it and you've lost anything mid-flow. No "draft"
of a Hunter.io verification queued, no "remind me later".

**Suggestion.** The modal should write a row to an `email_hunt_queue`
table the moment the user opens it, so partial work is preserved.
The nightly cron (already wired into the pipeline) then processes
the queue. The UI promises "Running now" (immediate) OR "Queued
for tonight's batch" (overnight), user chooses.

## 11. Voice reference editor has no "preview what Opus would write"

**Observation.** On /templates, you can now edit founder_bio and
voice_reference_email in the Voice Reference card. Saving works.
But there's no way to test the edit other than clicking Redraft on
a paragraph, waiting 5s, and reading the output. Feedback loop is too
long.

**Suggestion.** Add a "Test draft with these inputs" button next to
Save in the Voice Reference card — shows a 2-3 sentence credibility
paragraph inline so the user can see the effect of a founder_bio
tweak immediately. No save required; just a read-only preview.

## 12. The approval return parser has no confidence / conflict UI

**Observation.** Step 2 on /approval parses a reply text into per-row
decisions. When it lands, it shows the Step 3 table with "✓ Approved"
/ "✗ Reject" per row. But if the parser matched 19 of 20 and got one
wrong (e.g. the approver wrote "ok for Felicis, but pass on AENU"
and the parser flipped them), there's no way to see the CONFIDENCE
of each match. The user either trusts the whole batch or reads 20
rows manually.

**Suggestion.** Show a per-row confidence score on Step 3 (already
computed by the Haiku parser, just not surfaced) + an inline edit
affordance ("Haiku thought this was 'approved' with 60% confidence
— click to change to rejected"). Turns a black-box parser into a
reviewable proposal.

## Information flow — what's working

1. **The hedged-knowledge frame is consistent.** Post-Opus-swap,
   every /templates draft, every /tracker/[id]/draft synthesis, and
   every /approval/test-replies draft response opens with "My
   understanding is that {{FIRM}}…" or similar. The voice is stable
   across surfaces.

2. **Bracket policy is enforced in three places.** buildPromptForSection
   has a HARD RULE block; draftSectionWithHaiku has a regex guard;
   auditSectionWithOpus re-checks. Defence in depth. A bad Opus
   sample can't reach email_templates.

3. **Data lineage is visible on partner profile.** The PROVENANCE
   card on /partner/[id] shows "Last synced 2026-04-23 05:30:05"
   and a pointer to the sync script. Users know this data isn't
   magical — it came from a real file at a real time.

4. **Honest empty states.** /review "No drafts at +2 for this
   campaign" is a good example — it names the missing prerequisite
   and points at the next action. Good product copy, not "No data".

5. **Conflict banner on /match.** "EBRD is already in Panatere at
   -2 Bounced (6d). Adding them to SkySails risks a double-ask…" —
   this is the kind of cross-workstream signal that prevents real
   mistakes. Rule 4 from the runbook enforced automatically.

## Priority order if I had 2 hours

1. **#1 (campaign identity)** — makes every other screen coherent.
2. **#9 (sends history on tracker rows)** — closes the "what did I
   actually send" loop.
3. **#7 (tracker sidebar → action panel)** — the highest-leverage
   surface change.
4. **#12 (approval parser confidence)** — prevents silent
   mis-parses.
5. **#2 (display_name)** — cheap rename, high cleanliness win.

The rest are polish; these 5 are structural.
